# 项目标准交付工作流

## 适用于下一次任何新项目的复用方法

日期：2026-03-21  
版本：v1.0

---

## 1. 为什么需要这份工作流

很多项目低效，不是因为开发慢，而是因为：

- 一开始没有统一定义业务目标
- 产品和技术没有共同语言
- 中间不断返工
- 最后只能交付“代码”，交不出“完整项目”

因此，下一次做任何项目，建议强制按这套闭环执行。

---

## 2. 标准阶段

### Phase 0：项目定义

产物：

- 一句话项目定义
- 目标用户
- 关键业务问题
- 成功标准

输出文档：

- `BRD 初稿`

### Phase 1：业务与产品收敛

产物：

- BRD
- PRD
- 核心页面与信息架构

输出文档：

- `BRD`
- `PRD`

### Phase 2：UI / UX 定义

产物：

- 页面结构
- 组件规范
- 状态流转
- 视觉层级

输出文档：

- `UI Design Spec`

### Phase 3：技术方案

产物：

- 架构图
- 组件职责
- 数据模型
- 接口边界
- 部署策略

输出文档：

- `Technical Design`

### Phase 4：实现与联调

产物：

- 前端
- 后端
- 数据链路
- 运维能力

输出要求：

- 代码可运行
- 关键路径可演示

### Phase 5：测试与验收

产物：

- 功能测试
- 集成测试
- 发布回归
- 风险清单

输出文档：

- `Test Plan / Acceptance`

### Phase 6：上线与交接

产物：

- 在线地址
- 仓库地址
- 运行说明
- 运维入口
- 已知边界

输出文档：

- `Release & Handoff`

---

## 3. 标准交付清单

每个项目都至少要交付这 7 类文件：

1. `BRD`
2. `PRD`
3. `UI Spec`
4. `Technical Design`
5. `Test / QA`
6. `Release & Handoff`
7. `Project Delivery Index`

推荐再补：

8. `Roadmap / Next Phase`
9. `Runbook / Operations`

## 5. 每一阶段必须回答的问题

### BRD 必答

- 为什么做？
- 为谁做？
- 解决什么业务问题？
- 不做会损失什么？

### PRD 必答

- 用户要完成什么任务？
- 首页要回答什么问题？
- 核心模块是什么？
- 状态如何流转？

### UI 必答

- 信息层级是什么？
- 主 CTA 是什么？
- 用户如何感知系统状态？

### 技术 必答

- 系统怎么部署？
- 数据怎么流动？
- 任务如何调度？
- 出错怎么办？

### 测试 必答

- 关键链路有没有跑通？
- 如何确认功能可用？
- 已知边界有哪些？

### 发布 必答

- 在线地址是什么？
- 谁应该访问哪个入口？
- 如何启动、排查、停止？

---

## 6. 交付验收门槛

一个项目只有同时满足以下条件，才算“完整交付”：

1. 有明确业务定义
2. 有产品结构文档
3. 有 UI 规范
4. 有技术方案
5. 有测试与验收说明
6. 有上线地址
7. 有交接入口

如果缺一项，这个项目就还只是“半成品”。

---

## 7. 复用时的最小动作

下一次做别的项目时，最少要先做这 5 步：

1. 复制本目录结构
2. 先写 `BRD`
3. 再写 `PRD`
4. 再落 `UI + Technical`
5. 实现后补 `QA + Release`

不要一开始就直接写代码。

---

## 8. 推荐协作方式

### 老板 / 业务负责人

输入：

- 商业目标
- 目标客群
- 决策问题

### 产品/方案负责人

输出：

- BRD
- PRD
- UI Spec

### 技术负责人

输出：

- Technical Design
- 部署与运维方案

### QA / 交付负责人

输出：

- 测试清单
- 验收结果
- 上线说明

---

## 9. 复用价值

复用这套方法的意义不是“多写文档”，而是：

- 降低返工
- 提高交付完整度
- 让项目可交接
- 让下一次新项目更快启动

这会真正提升团队效率。
